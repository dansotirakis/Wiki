DDD

Domain Driven Design - desenho orientado a dôminio

  • Conceitos

    • Uma abordagem de desenho de software disciplinada que reúne um conjunto de conceitos, técnicas e princípios com foco no domínio e na lógica.
    • Esforço dedicado na camada de modelo que representa toda a complexidade das regras de negócio.
    • A equipe deve possuir um domain expert é um membro da equipe, profundo conhecedor do negócio.
    • Trata-se de uma regra de 3 partes contexto, problema e solução.
    • Este modo de desenho de software deve ser utilizado quando sistema posssui uma regra de negócio rica, e não se trata apenas de CRUDS
    • Contexto limitado por dominio, dentro deste limite todos os termos, definições e conceitos formam a linguagem ubíqua

  • Camadas

    User interface Application Domain Infrastructure Query
    Front-end Coordena tarefas e delega trabalhos Regras de negócio Recursos técnicos como persistência de dados, migrations etc... CQRS

  • JPA

    • AttributeConverter [Interface - Classe atributo]

      • convertToDatebaseColumn : Recebemos o nosso objeto instanciado por parâmetro e vamos apenas pegar o valor que vai para o banco de dados
      • convertToEntityAttribute : Recebemos o valor do banco de dados, e vamos instanciar um novo objeto do tipo Document

    • Atributo - Objeto [@Convert - Entity]

      • Ex :
        @Convert(converter = DocumentConverter.class)
        private Document documentNumber;
